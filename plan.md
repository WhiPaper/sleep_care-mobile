# Sleep Care Project Plan

## 1. 프로젝트 개요

### 1.1 프로젝트 목적
- 수험생의 학습 중 졸음 상태를 감지한다.
- 사용자의 실제 수면 패턴을 기록하고 분석한다.
- 졸음 데이터와 수면 데이터를 함께 분석하여 더 나은 수면 스케줄을 제안한다.
- 사용자가 스마트폰 앱에서 자신의 상태를 쉽게 확인하고 행동으로 옮길 수 있게 한다.

### 1.2 핵심 사용자
- 장시간 책상에서 공부하는 수험생
- 수면 부족 또는 불규칙한 생활 패턴으로 학습 효율이 떨어지는 사용자
- 시험 일정에 맞춰 수면 습관을 조정하고 싶은 사용자

### 1.3 핵심 가치
- 실시간 졸음 감지
- 수면 패턴 기반 분석
- 시험 일정 및 학습 계획 기반 수면 스케줄 제안
- 디바이스 간 연동을 통한 일상 밀착형 케어

## 2. 전체 제품 구성

본 프로젝트는 크게 3개의 프로그램으로 구성된다.

### 2.1 라즈베리파이5 프로그램
- 카메라를 통해 사용자의 얼굴 상태를 감지
- AI 모델을 활용해 졸음 여부를 판단
- 졸음 감지 시 LCD 및 스피커로 즉시 피드백 제공
- 감지 결과를 스마트폰 앱으로 블루투스 전송
- 탁상시계 형태의 임베디드 기기로 동작

### 2.2 스마트폰 앱
- 라즈베리파이에서 전송된 졸음 기록 수집
- 스마트워치에서 전송된 수면 기록 수집
- 두 데이터를 통합 분석
- 사용자의 학습 플랜과 시험 날짜를 반영해 수면 스케줄 제안
- 시각화된 리포트와 생활 가이드 제공

### 2.3 스마트워치 앱
- 사용자의 수면 관련 데이터 수집
- 수집한 데이터를 스마트폰 앱으로 전송
- 현재 단계에서는 졸음 감지는 제외
- 향후 확장 시 수면 외 생체 신호 활용 가능성 검토

## 3. 제품 시나리오

### 3.1 기본 사용 흐름
1. 사용자는 스마트폰 앱에서 계정 또는 로컬 프로필을 생성한다.
2. 사용자는 라즈베리파이 디바이스와 스마트워치를 스마트폰 앱에 연결한다.
3. 라즈베리파이는 공부 중 사용자의 졸음 상태를 감지한다.
4. 스마트워치는 수면 데이터를 기록한다.
5. 스마트폰 앱은 두 기기의 데이터를 수집하고 저장한다.
6. 앱은 학습 플랜, 시험 날짜, 졸음 빈도, 수면 패턴을 종합 분석한다.
7. 사용자에게 수면 스케줄과 생활 가이드를 제안한다.

### 3.2 주요 사용자 가치 시나리오
- 공부 중 졸음이 자주 발생하면 앱이 수면 부족 가능성을 알려준다.
- 사용자가 시험 날짜를 입력하면 시험 주간에 맞춰 취침 및 기상 시간을 조정하는 계획을 제안한다.
- 사용자가 학습 플랜을 입력하면 공부 시간대에 맞는 수면 시간과 휴식 시간 가이드를 제안한다.

## 4. 시스템 구조 초안

### 4.1 데이터 흐름
1. 라즈베리파이 카메라 입력
2. 라즈베리파이 AI 졸음 판별
3. 졸음 이벤트 생성
4. 블루투스로 스마트폰 앱 전송
5. 스마트워치 수면 데이터 수집
6. 스마트폰 앱으로 동기화
7. 모바일 앱 내 통합 저장 및 분석
8. 사용자에게 리포트 및 수면 스케줄 제안

### 4.2 기기별 역할 분리
- 라즈베리파이: 실시간 감지와 즉시 반응
- 스마트워치: 수면 데이터 수집
- 스마트폰 앱: 데이터 허브, 분석 엔진, 사용자 인터페이스

### 4.3 통신 구조 초안
- 라즈베리파이 ↔ 스마트폰 앱: Bluetooth
- 스마트워치 ↔ 스마트폰 앱: 스마트워치 플랫폼 기본 연동 방식 사용
- 스마트폰 앱 내부: 로컬 저장소 우선, 필요 시 향후 클라우드 확장 검토

### 4.4 모바일 앱 기술 스택 방향
- 앱 본체: Kotlin
- UI: Jetpack Compose + Material 3
- 아키텍처: MVVM + Repository + UDF(단방향 데이터 흐름)
- 상태관리 / 비동기 / DI: ViewModel + Kotlin Coroutines + Flow + Hilt
- 로컬 저장: Room + DataStore
- 백그라운드 작업: WorkManager
- 라즈베리파이 통신: BLE(GATT) 기준으로 설계
- 초기 페어링: CompanionDeviceManager 우선 검토
- 갤럭시 워치 수면 데이터: 1차 선택은 Health Connect

### 4.5 모바일 앱 기술 스택 메모
- Android 공식 권장 방향에 맞춰 UI 레이어와 Data 레이어를 분리한다.
- Repository와 ViewModel, Coroutines/Flow 조합으로 BLE 수신 상태, 수면 기록 조회, 추천 결과 계산, 화면 상태 갱신을 분리한다.
- Compose는 현재 Android 권장 UI 툴킷이므로 신규 화면은 Compose 기준으로 설계한다.
- Room은 졸음 이벤트, 수면 세션, 시험 일정, 학습 플랜, 추천 결과 캐시에 사용한다.
- DataStore는 설정값, 마지막 동기화 시각, 연결된 기기 정보, 사용자 선호값 저장에 사용한다.
- WorkManager는 주기적 동기화, 리포트 계산, 재시도 작업처럼 앱 재시작 이후에도 이어져야 하는 작업에 사용한다.

### 5-A.1 목표
- 사용자의 졸음 상태를 실시간으로 감지
- 졸음 발생 시 즉시 깨우기 또는 주의 환기

### 5-A.2 주요 기능
- 카메라 영상 수집
- 얼굴 인식 또는 얼굴 영역 추적
- 눈 감김, 고개 숙임, 깜빡임 패턴 등 기반 졸음 판단
- LCD 상태 표시
- 스피커 알림 또는 경고음 재생
- 블루투스 페어링 및 데이터 전송
- 기기 상태 확인용 간단한 UI 제공

### 5-A.3 세부 모듈
- 영상 입력 모듈
- AI 추론 모듈
- 졸음 이벤트 판단 모듈
- 알림 출력 모듈
- Bluetooth 통신 모듈
- 기기 설정 및 로그 모듈

### 5-A.4 구현 단계 제안
1. 카메라 입력 안정화
2. 얼굴 및 눈 상태 감지 기초 구현
3. 단순 규칙 기반 졸음 감지 MVP 구현
4. LCD 출력 및 경고음 연결
5. Bluetooth 데이터 전송 구현
6. 실제 환경 테스트 후 AI 모델 정교화

### 5-A.5 고려 사항
- 조명 변화와 안경 착용 여부에 대한 대응
- 장시간 실행 시 발열 및 성능 안정성
- 오탐지와 미탐지 균형 조정
- 오프라인 환경에서도 동작 가능해야 함

### 5-A.6 내 의견
- 초기 버전에서는 복잡한 대형 모델보다 경량 모델 또는 규칙 기반 판별을 우선하는 것이 현실적이다.
- 졸음 감지 기준은 처음부터 완벽하게 맞추기 어렵기 때문에, 사용자별 보정 기능을 후속 과제로 두는 것이 좋다.
- 스피커는 꼭 넣는 편이 좋다. LCD만으로는 공부 중 즉각적인 각성 효과가 약할 가능성이 높다.

## 5-B. 스마트폰 앱 계획

### 5-B.1 목표
- 전체 서비스의 중심 허브 역할 수행
- 기기 데이터 수집, 저장, 분석, 시각화
- 사용자 맞춤 수면 스케줄 제안 제공

### 5-B.2 주요 기능
- 라즈베리파이 디바이스 연결 및 상태 확인
- 스마트워치 데이터 연동
- 졸음 기록 타임라인 조회
- 수면 기록 조회
- 통합 분석 리포트
- 수면 스케줄 제안
- 학습 플랜 입력 기능
- 시험 날짜 입력 기능
- 푸시 알림 또는 리마인더

### 5-B.3 수면 스케줄 제안 기능

#### 학습 플랜 기반 제안
- 사용자가 공부 시간표 또는 학습 플랜을 입력
- 집중 공부 시간대와 기상 시간을 고려해 권장 취침 시간 제안
- 낮 시간 졸음이 잦은 경우 학습 플랜 내 휴식 배치도 함께 제안

#### 시험 날짜 기반 제안
- 사용자가 시험 날짜를 입력
- 시험 직전 며칠 또는 몇 주 동안 목표 기상 시간에 맞춰 수면 리듬 조정 제안
- 시험 당일 컨디션 최적화를 위한 취침 시간 가이드 제공

#### 통합 추천 로직 초안
- 최근 수면 시간
- 수면 시작/종료 시각의 규칙성
- 공부 중 졸음 빈도
- 시험 일정까지 남은 기간
- 사용자가 원하는 기상 시간

### 5-B.4 화면 구조 초안
- 온보딩
- 기기 연결 화면
- 홈 대시보드
- 졸음 분석 화면
- 수면 분석 화면
- 수면 스케줄 제안 화면
- 학습 플랜 입력 화면
- 시험 일정 관리 화면
- 설정 화면

### 5-B.5 세부 모듈
- Bluetooth 연동 모듈
- 워치 데이터 동기화 모듈
- 로컬 데이터 저장 모듈
- 분석 엔진
- 추천 엔진
- UI/UX 모듈
- 알림 모듈

### 5-B.5-1 권장 기술 스택
- Kotlin
- Jetpack Compose + Material 3
- MVVM + Repository + UDF(단방향 데이터 흐름)
- ViewModel + Kotlin Coroutines + Flow + Hilt
- Room + DataStore
- WorkManager
- BLE(GATT)
- CompanionDeviceManager 우선 검토
- Health Connect 우선 검토

### 5-B.5-2 저장소 역할 분리
- Room: 졸음 이벤트, 수면 세션, 시험 일정, 학습 플랜, 추천 결과 캐시
- DataStore: 설정값, 마지막 동기화 시각, 연결된 기기 정보, 사용자 선호값

### 5-B.5-3 통신 및 동기화 기준
- 라즈베리파이와의 연결은 Android BLE 중앙 역할 기준으로 설계한다.
- 데이터 송수신은 GATT 서비스/특성 구조로 정의한다.
- 초기 기기 탐색과 페어링 UX는 CompanionDeviceManager 우선으로 검토한다.
- 워치 수면 데이터는 1차적으로 Health Connect 연동 가능성을 기준으로 설계한다.

### 5-B.6 구현 단계 제안
1. 모바일 앱 기본 구조 및 화면 골격 구성
2. 라즈베리파이 데이터 수신 기능 구현
3. 스마트워치 데이터 수신 기능 구현
4. 데이터 저장 구조 설계
5. 기초 리포트 및 시각화 구현
6. 수면 스케줄 제안 알고리즘 MVP 구현
7. 학습 플랜 및 시험 일정 반영 고도화

### 5-B.7 고려 사항
- 분석 결과는 설명 가능해야 사용자가 신뢰하기 쉽다.
- 너무 많은 정보를 보여주기보다 행동으로 이어질 수 있는 제안이 중요하다.
- 앱이 단순 기록 도구가 아니라 실제 생활 루틴 코치처럼 느껴지게 설계하는 것이 좋다.

### 5-B.8 내 의견
- 스마트폰 앱은 이 프로젝트의 핵심 제품이므로 가장 먼저 사용자 경험 관점에서 설계해야 한다.
- 초기 추천은 정교한 AI보다 규칙 기반 추천으로 시작해도 충분하다.
- 홈 화면에서는 "어제 수면 상태", "최근 졸음 빈도", "오늘 추천 취침 시간" 3가지를 핵심 카드로 우선 노출하는 구성이 좋아 보인다.

### 5-C.1 목표
- 사용자의 수면 데이터를 안정적으로 수집
- 스마트폰 앱으로 전달 가능한 최소 기능 구현

### 5-C.2 주요 기능

- 향후 확장을 고려한 센서 데이터 구조 설계

### 5-C.3 구현 범위

  - 수면 시간 기록

- 후속 단계
  - 수면 단계 정보 활용 검토
  - 심박 등 추가 데이터 활용 검토

### 5-C.4 구현 단계 제안
1. 대상 워치 플랫폼 확정
2. 수면 데이터 접근 가능 여부 확인
3. 수집 가능한 최소 데이터 정의

- 백그라운드 동기화 제한
- 배터리 소모 최소화

### 5-C.6 내 의견
- 워치 앱은 처음부터 범위를 넓히기보다 "수면 데이터 안정 수집" 하나에 집중하는 것이 좋다.
- 실제 구현 난이도는 워치 플랫폼 제약에 크게 좌우되므로, 초기에 플랫폼 확정이 매우 중요하다.

## 6. 데이터 설계 초안

### 6.1 핵심 데이터 항목

#### 졸음 이벤트 데이터
- 발생 시각

- 기기 ID

- 수면 품질 관련 지표
- 동기화 시각

#### 사용자 입력 데이터

- 시험 날짜
- 목표 기상 시간
- 선호 취침 시간대

### 6.2 분석 결과 데이터

- 주간 수면 규칙성

## 7. 분석 및 추천 로직 방향

### 7.1 초기 버전
- 규칙 기반 분석
- 최근 며칠간 수면 부족 여부 계산
- 졸음 발생 시간대와 수면 시간의 상관관계 단순 분석
- 시험일과 학습 플랜을 고려한 권장 취침 시간 제안

### 7.2 고도화 방향
- 개인화 모델 적용
- 사용자 반응 기반 추천 조정
- 장기 패턴 분석
- 컨디션 예측 기능 검토

### 7.3 추천 예시
- 최근 3일 평균 수면이 부족하고 오후 졸음이 자주 발생하면 더 이른 취침 시간을 제안
- 시험 날짜가 가까워질수록 시험 시간에 맞춘 기상 리듬 조정을 제안
- 아침 공부 플랜이 많으면 야간 취침 시간을 더 일찍 제안

## 11. 협업용 논의 카테고리

이 섹션은 Notion에서 그대로 옮겨 협업용 논의 보드 또는 문서 구조로 사용할 수 있다.

### 11.1 제품 기획
- 우리 서비스의 핵심 문제 정의는 무엇인가
- 수험생이 실제로 가장 불편해하는 지점은 무엇인가
- 단순 기록 앱이 아니라 어떤 행동 변화를 만들고 싶은가
- GPT 의견: 이 프로젝트의 가장 강한 차별점은 "공부 중 졸음"과 "실제 수면"을 함께 본다는 점이다.
- GPT 의견: 단순한 수면 기록 앱처럼 보이기보다 학습 효율 개선 서비스라는 메시지를 분명히 잡는 것이 좋다.

### 11.2 사용자 경험
- 사용자가 앱을 열었을 때 가장 먼저 봐야 하는 정보는 무엇인가
- 졸음 경고는 어느 수준에서 울려야 부담이 적고 효과적인가
- 수면 스케줄 제안은 얼마나 구체적으로 보여줄 것인가
- GPT 의견: 홈 화면은 "어제 수면 상태", "최근 졸음 빈도", "오늘 추천 취침 시간"을 우선 보여주는 구성이 적합해 보인다.
- GPT 의견: 학습 플랜 기반 제안은 유용하지만 입력 피로도가 생길 수 있으므로 UX를 최대한 단순화하는 것이 좋다.

### 11.3 라즈베리파이 디바이스
- 카메라 위치와 시야각은 어떻게 설계할 것인가
- LCD는 어떤 편의기능을 넣을 것인가
- 스피커를 넣을 경우 어떤 방식의 알림이 적절한가
- GPT 의견: 라즈베리파이를 탁상시계처럼 만드는 방향은 매우 좋고 실제 책상 환경에 자연스럽게 놓일 수 있어야 한다.
- GPT 의견: 스피커는 넣는 편이 좋다. LCD만으로는 공부 중 즉각적인 각성 효과가 약할 수 있다.

### 11.4 모바일 앱
- 핵심 화면 우선순위는 무엇인가
- 분석 결과를 어떤 그래프로 보여줄 것인가
- 학습 플랜 입력 UX는 자유 입력, 캘린더형, 시간표형 중 무엇이 좋은가
- GPT 의견: 스마트폰 앱은 이 프로젝트의 핵심 허브이므로 가장 먼저 사용자 경험 관점에서 설계하는 편이 좋다.
- GPT 의견: 분석 결과는 많이 보여주는 것보다 실제 행동으로 옮길 수 있는 제안을 중심으로 구성하는 것이 좋다.

### 11.5 워치 앱
- 어떤 워치 플랫폼을 1차 대상으로 할 것인가
- 수면 데이터는 자체 측정인지 플랫폼 API 연동인지 결정이 필요한가
- GPT 의견: 워치 앱은 처음부터 범위를 넓히기보다 수면 데이터 안정 수집에 집중하는 편이 현실적이다.
- GPT 의견: 실제 구현 난이도는 워치 플랫폼 제약에 크게 좌우되므로 초기에 대상 플랫폼을 먼저 확정하는 것이 중요하다.

### 11.6 데이터 및 분석
- 졸음과 수면의 상관관계를 어떤 방식으로 보여줄 것인가
- 추천 결과에 대한 설명 문구는 어떻게 만들 것인가
- 개인화 수준을 어디까지 허용할 것인가
- GPT 의견: 초기 추천은 정교한 AI보다 규칙 기반 추천으로 시작해도 충분하다.
- GPT 의견: 추천 결과는 왜 그런 제안이 나왔는지 설명 가능해야 사용자가 신뢰하기 쉽다.

### 11.7 개발 운영
- 역할 분담은 어떻게 나눌 것인가
- MVP 기준은 어디까지로 볼 것인가
- 테스트는 어떤 환경에서 우선 수행할 것인가
- GPT 의견: MVP는 "졸음 감지", "수면 기록 수집", "간단한 수면 제안"까지만 성공해도 충분히 의미 있다.
- GPT 의견: 너무 이른 단계에서 정교한 AI 추천까지 포함하면 개발 난이도가 급격히 올라갈 수 있다.
